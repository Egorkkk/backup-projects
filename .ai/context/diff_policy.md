# diff_policy.md

## Главный принцип

Предпочитать минимальный, обозримый и предсказуемый diff.

---

## Что считается хорошим diff

Хороший diff:
- решает одну задачу
- не затрагивает unrelated files
- не переименовывает структуру без нужды
- не включает "заодно" рефакторинг
- легко review-ится человеком

---

## Обязательные правила

1. Не переписывать файл целиком, если достаточно точечного изменения.
2. Не перемещать код между слоями без необходимости.
3. Не переименовывать публичные сущности без явной причины.
4. Если изменение затрагивает >5 файлов, объяснить почему.
5. Сначала перечислить affected files для нетривиальных задач.
6. Отдельно помечать рискованные изменения.

---

## Когда допустим более крупный diff

Более крупный diff допустим, если:
- создаётся новая подсистема с нуля
- нужно провести обязательную нормализацию структуры
- без этого невозможно соблюсти архитектурную границу
- это часть заранее согласованного design-first шага

---

## Что запрещено

- hidden cleanup
- hidden style-only rewrite of many files
- silent public API change
- silent config format change
- merge нескольких задач в один diff

---

## Практическое правило

Если можно выбрать между:
- более "красивым" крупным изменением
- более локальным и безопасным изменением

нужно выбирать локальное и безопасное.
