# AGENTS.md

## Назначение

Этот репозиторий разрабатывается в режиме "один Codex-агент + разные prompt-файлы + ручное управление задачами человеком".

Цель:
- двигаться строго по задачам из `TASKS.md`
- не нарушать критический путь из `ROADMAP.md`
- сохранять предсказуемую архитектуру
- минимизировать хаос от ИИ-генерации
- делать по одной атомарной задаче за раз

---

## Source of truth

При любых сомнениях приоритет такой:

1. `ROADMAP.md`
2. `TASKS.md`
3. `ARCHITECTURE.md`
4. `DEFINITION_OF_DONE.md`
5. этот файл (`AGENTS.md`)

Если реализация противоречит roadmap или tasks, реализация неверна.

---

## Рабочая модель

Используется один Codex-агент.

Режим работы задаётся через prompt-файлы из `.ai/prompts/`:

- `design_contracts.md` — проектирование DTO, сигнатур, модулей, структуры
- `implement_task.md` — реализация атомарной задачи
- `review_changes.md` — архитектурное и инженерное ревью
- `write_tests.md` — написание тестов
- `debug_issue.md` — разбор и исправление бага
- `refactor_safe.md` — локальный безопасный рефакторинг

## Основные правила

### 1. Одна атомарная задача за раз
Не брать несколько логически связанных задач одновременно.
Одна задача должна иметь:
- узкий scope
- понятный результат
- ограниченное количество файлов
- тестируемый результат

### 2. Сначала контракты, потом реализация
Для важных подсистем сначала фиксируются:
- DTO
- enums
- сигнатуры
- интерфейсы модулей
- структура файлов

И только потом пишется реализация.

### 3. Сначала CLI, потом Web UI
Если фича имеет и CLI, и web-представление:
- сначала реализовать через services + CLI
- только потом добавлять web route / template / UI action

### 4. Не смешивать слои
Запрещено:
- web-коду знать SQL/SQLite напрямую
- CLI-коду ходить в SQL напрямую
- jobs-коду знать HTML/templates
- services-коду знать Flask request/response

### 5. Минимальный diff
Нужно предпочитать:
- точечные изменения
- минимальные патчи
- сохранение существующей структуры

Нельзя без причины:
- массово переименовывать файлы
- переписывать файл целиком
- делать скрытый рефакторинг вне scope

### 6. Человек вручную управляет статусами задач
Codex НЕ обновляет `TASKS.md` автоматически.
Статусы задач меняет человек вручную после проверки.

## Границы v1

В v1 входит:
- config loader
- SQLite registry
- roots discovery
- structural scan
- incremental scan
- manual includes
- policy engine
- manifest builder
- restic adapter
- backup flow
- daily/weekly jobs
- reports/export
- Flask UI
- cron-ready CLI pipeline

В v1 не входят:
- systemd timers как основной режим
- restore verify
- сложные эвристики rename/move
- расширенный file review UI
- retention automation beyond baseline
- продвинутая оптимизация scanning logic до базовой рабочей версии

---

## Ограничения по изменениям

Codex обязан:
- перечислить изменяемые файлы до начала реализации, если задача нетривиальна
- явно указать, что НЕ делается в рамках задачи
- не расширять scope без необходимости
- не добавлять новые зависимости без объяснения
- не менять публичные контракты без явной причины

Если задача тянет за собой архитектурный сдвиг, надо сначала перейти в режим `design_contracts.md`.

## Когда нужно использовать review отдельно

Отдельный запуск `review_changes.md` нужен обязательно, если:
- изменено больше 5 файлов
- затронуты `services/`, `adapters/` и `web/` одновременно
- меняются DTO или enums
- меняются DB schema / repo contracts
- меняется CLI public behavior

---

## Когда нужно сначала использовать design_contracts

Сначала `design_contracts.md`, потом реализация — обязательно для:
- config
- db schema
- repositories
- domain DTO
- policy engine
- manifest builder
- jobs orchestration
- report formats
- web routes backed by service contracts

---

## Запрещённое поведение

Codex не должен:
- перескакивать через критический путь
- реализовывать UI раньше готового service/CLI слоя
- добавлять "полезный бонусный функционал" вне задачи
- менять unrelated files
- делать магические оптимизации до рабочей baseline-версии
- silently менять naming conventions
- silently менять формат конфигов
- silently менять контракты CLI

---

## Практическое правило

Если есть сомнение:
- делать меньше
- делать проще
- фиксировать контракт
- оставлять TODO явно
- не угадывать лишнее
