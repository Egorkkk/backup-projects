# AI_WORKFLOW.md

## Цель

Этот файл описывает практический порядок работы с одним Codex-агентом в VS Code.

---

## Базовый цикл работы

### Шаг 1. Взять следующую задачу
Человек получает следующую задачу из `TASKS.md` в правильном порядке.

### Шаг 2. Определить режим
Выбрать один из режимов:
- design contracts
- implement task
- review changes
- write tests
- debug issue
- safe refactor

### Шаг 3. Ограничить scope
Перед запуском Codex задача должна быть сформулирована как атомарная:
- один модуль
- или один контракт
- или одна CLI-команда
- или одна таблица + repo
- или один service + tests

### Шаг 4. Сначала контракт, если нужно
Если затрагиваются:
- DTO
- enums
- schema
- repositories
- service boundaries
то сначала используется `.ai/prompts/design_contracts.md`.

### Шаг 5. Реализация
После фиксации контракта используется `.ai/prompts/implement_task.md`.

### Шаг 6. Ревью
Если задача не совсем тривиальна, запускается `.ai/prompts/review_changes.md`.

### Шаг 7. Тесты
Если тесты ещё не написаны или неполны, используется `.ai/prompts/write_tests.md`.

### Шаг 8. Ручная проверка
Человек:
- смотрит diff
- запускает тесты / smoke checks
- вручную отмечает задачу в `TASKS.md`

---

## Рекомендуемый порядок prompt-ов

### Для новой подсистемы
1. `design_contracts.md`
2. `implement_task.md`
3. `review_changes.md`
4. `write_tests.md`

### Для маленькой локальной задачи
1. `implement_task.md`
2. `review_changes.md`
3. `write_tests.md`

### Для бага
1. `debug_issue.md`
2. `review_changes.md`
3. `write_tests.md`

### Для точечного улучшения
1. `refactor_safe.md`
2. `review_changes.md`

---

## Как формулировать задачу Codex

Каждая задача должна содержать:

- цель
- scope
- явные ограничения
- список ожидаемых файлов
- список того, что НЕ входит в задачу
- критерии готовности
- команду(ы) проверки, если известны

---

## Как делить чаты / сессии

Рекомендуется:
- один чат = одна атомарная задача
- отдельный чат для дебага
- отдельный чат для проектирования новой подсистемы
- отдельный чат для review/refactor

---

## Когда не надо просить Codex писать код сразу

Нужно остановиться на контрактном дизайне, если:
- неясна структура DTO
- неясно, где должен жить модуль
- есть риск смешивания слоёв
- есть риск преждевременного UI
- есть несколько возможных DB/API контрактов

---

## Политика ручного контроля

Человек вручную:
- выбирает следующую задачу
- подтверждает крупные контракты
- решает, принимать ли diff
- обновляет `TASKS.md`
- принимает решение о merge

Codex не является источником архитектурной истины.
Codex — исполнитель в рамках уже определённых границ.

---

## Предпочтительный стиль изменений

Предпочтительно:
- минимальный diff
- точечные изменения
- ясные имена
- простая baseline-реализация
- явные TODO вместо лишней магии

Нежелательно:
- переписывание половины проекта ради "красоты"
- premature optimization
- смешение проектирования и реализации в большой куче
- массовое изменение структуры репозитория без нужды

---

## Формат "готово"

Задача считается готовой, когда:
- выполнен scope
- нет скрытого расширения scope
- есть понятный отчёт
- пройдены нужные проверки
- архитектурные границы не нарушены
- человек принял результат
