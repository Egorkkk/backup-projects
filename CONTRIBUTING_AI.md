# CONTRIBUTING_AI.md

## Общий принцип

ИИ работает в этом репозитории как аккуратный исполнитель, а не как автор новой архитектуры "с нуля".

---

## Обязательные правила

1. Делать только то, что входит в scope текущей задачи.
2. Не менять unrelated files.
3. Не делать скрытых рефакторов.
4. Не добавлять зависимости без явного объяснения.
5. Не менять public contracts без причины.
6. Не смешивать слои.
7. Сначала проектировать контракты, потом писать реализацию.
8. Сначала CLI/service, потом web UI.
9. Предпочитать minimal diff.
10. Оставлять явные TODO вместо догадок.

---

## Что считать хорошим результатом

Хороший результат:
- локален
- понятен
- тестируем
- не ломает архитектуру
- легко review-ится человеком
- не создаёт лишней связности

---

## Что считать плохим результатом

Плохой результат:
- "заодно" меняет 8 соседних модулей
- перетаскивает код между слоями без нужды
- пишет UI до service layer
- тащит SQL в CLI/web
- подменяет baseline сложной магией
- делает красивые, но лишние абстракции

---

## Dependency policy

Добавлять новую зависимость можно только если:
- без неё решение заметно хуже
- она действительно нужна в текущем этапе
- она не нарушает простоту локального dev setup
- в отчёте явно указано, зачем она добавлена

---

## File change policy

Если меняется больше 5 файлов, нужно:
- объяснить зачем
- перечислить affected files
- показать, почему это не scope creep

---

## Naming policy

Не переименовывать:
- модули
- DTO
- enums
- команды CLI
- ключи конфигов

если это не требуется задачей явно.

---

## Reporting policy

После каждой задачи обязательно сообщать:
- что сделано
- какие файлы изменены
- что не делалось намеренно
- какие проверки выполнены
- какие риски / TODO остались
